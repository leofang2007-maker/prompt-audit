# Claude Code adapter

Reports every prompt submitted in [Claude Code](https://code.claude.com) to a prompt-audit server, via
a `UserPromptSubmit` hook. Pure audit: **non-blocking and fail-open** ‚Äî it never delays or rejects a
prompt, and any error passes through silently.

The headline feature is **enterprise enforcement**: Claude Code's `managed-settings.json` has the
highest settings precedence and *cannot be overridden by users*. With `allowManagedHooksOnly: true`,
every user/project hook is blocked, so the audit hook is pushed via MDM and the developer **cannot
remove or disable it**. (Design: [spec 0005](../../docs/specs/0005-claude-code-adapter.md).)

## What it captures

Claude Code's hook JSON has `session_id`, `prompt`, `cwd`, `transcript_path`, `prompt_id`,
`permission_mode` ‚Äî but **no user identity and no timestamp**. The shell hook fills those in:

| ingest field | source |
|---|---|
| `prompt`, `session_id`, `cwd`, `transcript_path` | the hook JSON |
| `timestamp` | generated by the hook (`date -u`) |
| `event_id` | `prompt_id` ‚Üí else `sha256(session_id \| minute \| prompt)` (idempotency) |
| `hostname` | `hostname` |
| `user_email`, `user_name` | env `PROMPT_AUDIT_USER_*` ‚Üí `git config user.email/name` ‚Üí OS user |
| `org_id`, `org_name`, `user_uid` | env `PROMPT_AUDIT_ORG_ID` / `_ORG_NAME` / `_USER_UID` |
| `repo`, `branch` | derived from the cwd's git (best-effort) |

## Prerequisites

`jq` and `curl` (pre-installed on most dev machines; if missing, the hook passes through). POSIX `sh`.

## Configuration

Set the ingest URL and the org's **ingest token** (write-only ‚Äî it cannot read the audit log). Env vars
win, else a `KEY=VALUE` config file is loaded from the first of:

1. `<cwd>/.prompt-audit.conf` ‚Äî per-project
2. `${PROMPT_AUDIT_DATA:-~/.prompt-audit}/config.conf` ‚Äî **best for MDM fleet rollout**
3. `~/.prompt-audit.conf` ‚Äî single machine
4. the bundled `config.conf` (placeholder only ‚Äî never commit a real token)

```sh
PROMPT_AUDIT_GATEWAY_URL=https://audit.your-domain.com/api/v1/prompts
PROMPT_AUDIT_GATEWAY_TOKEN=iat_your_org_ingest_token
```

## Install ‚Äî single user (not enforced)

Merge `settings/settings.json` into `~/.claude/settings.json`, replacing `<ADAPTER_DIR>` with the
absolute path to this directory. Then set config (above). Verify with a prompt, then check the audit UI.

## Install ‚Äî enterprise (enforced, users cannot remove)

Deploy `settings/managed-settings.json` (replace `<ADAPTER_DIR>`) to the managed path for each OS, via
your MDM / config-management (Jamf, Intune, Ansible, ‚Ä¶):

| OS | Managed path |
|---|---|
| macOS | `/Library/Application Support/ClaudeCode/managed-settings.json` |
| Linux / WSL | `/etc/claude-code/managed-settings.json` |
| Windows | `C:\Program Files\ClaudeCode\managed-settings.json` |

Also push the adapter scripts to `<ADAPTER_DIR>` and the config (URL + token) to
`~/.prompt-audit/config.conf` (or the env). `allowManagedHooksOnly: true` means only managed hooks run
‚Äî user/project hooks are blocked, so the audit hook is guaranteed and un-removable. This pairs with
[coverage detection](../../docs/specs/0004-reporting-coverage-gap-detection.md): on a managed fleet, a
silent host is a real anomaly, not a missing install.

## Alternative ‚Äî native HTTP hook (zero script)

`settings/managed-settings.http.json` uses Claude Code's built-in `type: http` hook to POST straight to
the ingest endpoint ‚Äî no shell script, no `jq`/`curl` dependency. Trade-off: **no identity enrichment,
no idempotency `event_id`**, and it sends Claude Code's raw JSON (`prompt`/`session_id`/`cwd`/
`transcript_path` map to our contract; `prompt_id`/`permission_mode` are ignored; no `hostname`). Use it
when you want the simplest possible deployment and don't need per-user attribution. Only allow-listed
env vars are interpolated into headers (`allowedEnvVars`), so the token isn't inlined.

## Security

- The only credential on the machine is the **write-only** per-org ingest token; a leak can't read a
  single audit record.
- The hook is **not covert** ‚Äî it runs visibly in the Claude Code transcript, and the server's public
  `/api/v1/transparency` endpoint discloses exactly what's captured. This is a security control, not
  surveillance (see [spec 0003](../../docs/specs/0003-anti-surveillance-guardrails.md)).
- Never commit a real token to the bundled `config.conf`; deliver it via MDM/env.

## Testing

Feed a sample event to the hook and watch the payload it builds:

```sh
echo '{"session_id":"s1","cwd":"'"$PWD"'","transcript_path":"/tmp/t.jsonl","prompt":"hello audit","prompt_id":"11111111-1111-1111-1111-111111111111","hook_event_name":"UserPromptSubmit"}' \
  | PROMPT_AUDIT_GATEWAY_URL=https://audit.your-domain.com/api/v1/prompts \
    PROMPT_AUDIT_GATEWAY_TOKEN=iat_... sh scripts/report-prompt.sh
```

Exit code is always 0. On success a row appears in the audit list; on failure the payload is spooled to
`~/.prompt-audit/failed/` and re-sent by `drain-failed.sh` at the next session start.
